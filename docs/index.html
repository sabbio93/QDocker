Sottoprogetto di ricerca nell’ambito del progetto QRobotEvitaMuri

Vision

La nostra software house possiede un linguaggio instanza di un metalinguaggio EBNF che permette di creare un modello del problema. Una delle caratteristiche cardine di questo linguaggio è il fatto di essere eseguibile e quindi presentabile come risultato al cliente.

La ricerca di un linguaggio formale per esprimere dipendenze e metodo di deploy delle applicazioni create con questo custom language ha dato il via a una ricerca a più ampio spettro che ha richiesto la creazione di un progetto di ricerca e sviluppo. Da questo progetto si spera poi partono diversi sotto progetti derivati dalle considerazioni sviluppate di seguito e spinti dalla spinta tecnologica che queste tecnologie stanno portando L’introduzione, infatti, di queste tecnologie ha creato e dato la spinta alla realizzazione di un insieme di nuove tecnologie e paradigmi di programmazione che hanno cambiato lo sviluppo di sistemi a livello mondiale. Il cambiamento talvolta è stato così significativo da poter essere slegato dalla tecnologia iniziale e diventare un concetto cardine del moderno sviluppo.

La nostra vision è di integrare i vantaggi portati da queste nuove tecnologie nel nostro linguaggio custom in modo da poter beneficiare dei nuovi costrutti, anche in modelli sviluppati in passato, cercando di conseguire un effort minimo se non nullo.

Problema iniziale

Il problema iniziale posto all'inizio del progetto è la ricerca ed eventuale implementazione di un linguaggio formale che permetta di descrivere:

  • Dipendenze software degli applicativi che andiamo a installare.
  • Dipendenze di ambiente degli applicativi.

In questo modo la messa in esecuzione del software risulta agevolata e automatizzabile.

Problemi derivanti dalla ricerca

Dalla ricerca condotta sul precedente problema sono però scaturite diverse tematiche di largo interesse per la nostra software house quali:

  • La possibilità di poter eseguire il codice su piattaforme eterogenee avendo un effort per gli sviluppatori minimo
  • Diminuire drasticamente i tempi di setup dei sistemi
  • Abilitare uno scenario di deploy distribuito automatizzato
  • Facilitare un futuro approccio DevOps 
  • Aggiungere la possibilità di sfruttare nel nostro linguaggio custom un modello di programmazione a microservizi oltre a quelli già presenti
  • Slegare da un linguaggio di programmazione l’implementazione del nostro linguaggio custom aprendo le porte ad applicazioni realizzate con più linguaggi di programmazione contemporaneamente

Analisi

Uno dei problemi che ci si può trovare ad affrontare quando  si intende introdurre e utilizzare in una software house un nuovo linguaggio è la configurazione dei sistemi. L’introduzione di nuovi linguaggi infatti può avere effetti non solo sugli sviluppatori interni all’azienda ma anche sui sistemisti che poi dovranno essere in grado di installare e eseguire le soluzioni create nel nuovo linguaggio magari in un contesto di contemporaneità con le soluzioni create con i linguaggi precedentemente utilizzati. La cosa si traduce spesso e  inevitabilmente non solo in un effort sostanziale a livello di tempo ma può aprire anche un insieme di tematiche legate alla coesistenza  di diversi ambienti. Esempi di soluzioni a questi problemi sono i risolutori di dipendenze come gradle e nmp che però non riescono ad agire a livello di sistema operativo e di rete e quindi si configurano, nel nostro caso come delle soluzioni parziali. La possibilità di poter “impacchettare” l’intera applicazione con le relative dipendenze sia di librerie che di sistema operativo sottostante è stata nel tempo risolta con le varie tecnologie di virtualizzazione e di containerizzazione. Inoltre, adottare una di queste soluzioni permetterebbe la creazione di un ambiente ove più linguaggi e tecnologie possono coesistere all’interno dello stesso host senza interferenze critiche. Si viene però in questo modo a creare un vincolo aggiuntivo ossia la necessità di uno scarso overhead da parte di queste soluzioni. Infatti lo sfruttare risorse eccessive potrebbe essere una tecnology killer in quanto richiederebbe investimenti in hardware che potrebbero non essere possibili. Imponendo invece uno scarso consumo di risorse si va in una direzione opposta in cui si può risparmiare risorse avendo una maggiore densità di ambienti di esecuzione sullo stesso hardware.

Fra le varie tecnologie che vanno a risolvere il nostro problema sono state individuate:

  • Le macchine virtuali
  • Le tecnologie con virtualizzazione a livello di SO come LXC e Docker

Le macchine virtuali sono una tecnologia ormai affermata che hanno permesso lo sviluppo di innumerevoli nuove tecnologie e nuovi paradigmi di sviluppo. In particolare per quanto riguarda le tematiche affrontate in questo progetto le VM(Macchine Virtuali) permettono:

  • di impacchettare l’intero ambiente di esecuzione e quindi slegarsi da tutte le dipendenze sia a livello di software che di SO(Sistema Operativo)
  • eseguire codice su diverse piattaforme eterogenee da comuni board a macro cluster(vedi anche Cloud)
  • abilitare uno scenario di deploy distribuito, automatizzato e orchestrato. A tal proposito le VM hanno dato la spinta iniziale per la creazione delle tecnologie di descrizione di deploy di cui ad esempio Vagrant e Chef sono un esempio
  • slegare dal linguaggio di implementazione l’intera applicazione in quanto le varie parti dell’applicazione possono essere distribuite tramite diverse VM creando uno dei primi approcci al SOAP

Inoltre le macchine virtuali hanno creato il primo approccio a un organizzazione DevOps permettendo agli sviluppatori di testare le applicazioni in locale o in cluster appositi senza la necessità di un intervento da parte degli amministratori di sistema. Il principale problema che ha sfavorito le VM nella scelta della tecnologia da impiegare nella nostra azienda sono le risorse impiegate da questa tecnologia. Le VM, infatti, dovendo impacchettare l’intero strato di SO sottostante all’applicazione necessitano di ingenti risorse che richiedono quindi un investimento non indifferente. Questo fatto inoltre sfavorisce un approccio alla composizione di applicazioni usando servizi fine grained in favore all’utilizzo di macro servizi per evitare ogni volta di replicare lo strato del SO.

La virtualizzazione a livello di Sistema Operativo è una tecnologia il cui sviluppo è stato iniziato nel 1982 con l’aggiunta di chroot ai sistemi operativi Linux. Questa tecnologia è poi derivate in molteplici altre innovazioni che nel tempo si sono susseguite per supportare la divisione di contesto a livello di sistema operativo. Negli ultimi tempi in particolare le due tecnologie derivate LXC e Docker hanno suscitato un notevole interesse e hype(fig 1) nelle comunità internazionali. Questo non solo per quello che realizzano queste tecnologie ma anche per tutt’un insieme di prodotti correlati che hanno permesso di creare e che talvolta hanno costituito un'innovazione perfino superiore alla tecnologia a container. Le tecnologie a container, infatti estendono le funzionalità delle macchine virtuali ma avendo prestazioni nettamente superiori permettono di creare servizi con una grana più fine chiamati microservizi. L’introduzione dei microservizi a così portata un nuovo approccio di programmazione e strutturazione delle applicazioni proprio come la rivoluzione delle macchine virtuali aveva porta la SOA(differenze fra microservizi e SOA). I microservizi sono una tecnologia talmente in auge da aver portato allo sviluppo di un linguaggio custom orientato ai microservizi di nome Jolie.

Date le migliori prestazioni e le nuove possibilità che i container permettono la scelta, non esclusiva, a cui, arrivati a questo punto, come analisti suggeriamo è proprio la tecnologia a container e in particolare nell'accezione dei Docker Container. Da sottolineare la non esclusività delle tecnologie in quanto queste tecnologie non solo non risentono di una contemporaneità di uso ad esempio insieme alle VM ma anzi ne giovano. Per questo motivo si vorrebbe nel tempo non solo introdurre all’interno della nostra software house la tecnologia a container docker ma anche altre diverse tecnologie a container, magari a diversi livelli di astrazione( LXC vs Docker), sia anche la tecnologia a Macchine Virtuali. L’introduzione di queste diverse tecnologie, sebbene possa comportare uno sforzo e un costo aggiuntivo iniziale, possono portare a un notevole risparmio in termini di tempo e denaro  molto maggiore allo sforzo iniziale una volta approntate. Per questo motivo si configurano a nostro parere come tecnologie essenziali per poter colmare i diversi abstraction gap individuati e competere nel mercato odierno potendo raggiungere una soluzione a complessità maggiori in un minore tempo.

Progetto

Di seguito andremo ad illustrare una implementazione della tecnologia venuta fuori dalla fase di ricerca. In particolare si è implementato la generazione automatica di un descrittore di ambiente, d’ora in poi chiamato dockerfile, nel linguaggio custom QA posseduto dalla nostra Software Factory.

Per questa prima versione consideriamo quindi solo di incapsulare i QActor all’interno di un Docker container che inferisce parte del contesto di esecuzione dal campo context dei QActor.

In versioni successive sarà possibile introdurre la definizione di Context, Actorop ,QActor, Plan in modo tale che l’insieme di queste informazioni vada a definire il dockerfile.

La tecnologia Xtext è stata precedentemente utilizzata per la realizzazione del Domain Specific Language QA e siccome la nostra intende essere un estensione sarà utilizzata anche da noi per la creazione automatica del dockerfile.

Per fare ciò creiamo e aggiungiamo alla precedente configurazione un generatore gendocker dove acquisiamo le informazioni di IP e Port dal campo Context definito in QActor e li usiamo per creare il nostro dockerfile.

class GenDocker {

def doGenerate(QActorSystemSpec system, String progName) {

  for (Context c : system.context)

{

               if (!c.standalone)

  {

             val jar = progName + "-1.0.jar"

             val zipDir = "./"

                 val zip = GenUtils.applicationPackageName(c.name) + ".Main"

+  c.name.toFirstUpper + "-1.0.zip"

     GenUtils.genFileDir(GenUtils.rootDir + "/docker/" + c.name, "", "Dockerfile", "",

'''

FROM java

RUN ["apt-get","install","unzip","-y"]

COPY ["«zipDir»«zip»","./"]

RUN ["unzip","./«zip»","-d","./"]

EXPOSE -P "«c.ip.port»"

CMD ["java","-jar","./«zip.substring(0,zip.lastIndexOf('.'))»/«jar»"]''')// «»

                        }

                }

        }

}

Prendendo qui ad esempio un progetto precedentemente realizzato coi QActor:

System helloworld

Context c ip [host="localhost" port=1234]

QActor a context c {

        Plan main normal

        delay time(1000);

        println("ciao")

}

Otterremo automaticamente il dockerfile:

FROM java

RUN ["apt-get","install","unzip","-y"]

COPY ["./it.unibo.computer.MainComputer-1.0.zip","./"]

RUN ["unzip","./it.unibo.computer.MainComputer-1.0.zip","-d","./"]

EXPOSE -P "1234"

CMD ["java","-jar","./it.unibo.computer.MainComputer-1.0/helloworld.jar"]

Sarà quindi sufficiente copiare nella destinazione il dockerfile e la cartella compressa creata ed eseguendo il comando:

docker build -t hello_world_app && docker run hello_world_app

automaticamente verrà creato il container e messa in esecuzione l’applicazione.

NOTA: Nel caso in cui sia necessario da parte dell QActor l’accesso a file specifici del pc (ad esempio la GPIO del RaspBerry) è necessario montarli esplicitamente nel comando di RUN. Questa operazione è momentaneamente demandata allo sviluppatore per essere poi implementata in maniera automatica in una futura versione del plugin.
